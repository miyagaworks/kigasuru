## 0. エグゼクティブサマリー

- 現行のフル入力（デフォルト＝上級者プリセット）は **1打あたり9〜11タップ・6画面遷移**。90打のラウンドで全打記録すると**約1,000タップ**になり、プレーファスト圧力下では構造的に継続不可能。
- 分析コード（analysis / dashboard）が実際に集計へ使っているのは **クラブ・着弾点(result)・ターゲット距離(distance)・ミスタイプ(missType) の4つだけ**。傾斜・ライ・強度・風・気温は**手動フィルタの絞り込み条件としてしか使われていない**。6ステップのうち4ステップは、現状の分析リターンに対して過剰投資である。
- 推奨案は「**ワンボード2タップ記録**」：クラブ→着弾点の2タップで即保存（保存ボタン廃止・Undo方式）。条件（傾斜・ライ・風）は「いつもと違う時だけ」チップで付与。コース・気温は**ラウンド単位**で1回だけ確定。距離はクラブ別実績から自動デフォルト。**平均2.2タップ/打（約80%削減）**で、現行分析の全機能（散布図・クラブ別精度・距離帯・ミス分析）を**一切損なわない**。
- ただし新UIの前に直すべき**出血**がある：オフライン同期は実質壊れている（Service Workerの同期関数が空実装、フォールバック側は全件再送＋重複量産＋ローカル削除）、距離未入力ショットはサーバーに永遠に保存されないのに成功表示が出る、位置・気温キャッシュに有効期限がなく過去ラウンドのコース名が今日の記録に付く。これらは記録データの信頼性をすでに毀損しており、Phase 0 として最優先で修正すべき（§4・§5）。

## 1. 現状分析

### 1.1 タップ数の実測（コードから計数）

入力レベルの初期値は `advanced`＝全項目ON（`app/settings/page.tsx:88`、`app/record/page.tsx:190-199` の enabledFields 初期値が全て true）。つまり**何も設定しない新規ユーザーは最大負荷フローから始まる**。

デフォルト（上級者）フローの1打、ステップごとの必要タップ数：

| ステップ | 操作 | タップ数 | 備考（実装箇所） |
|---|---|---|---|
| 1. 傾斜 | 8方向セグメント＋中央「平坦」から1つタップ | 1 | タップで自動遷移（`record/page.tsx:758-761`） |
| 2. ライ | 6択から1つ | 1 | 自動遷移 |
| 3. クラブ | 17本（デフォルト）から1つ | 1 | 自動遷移。3列×6行が `max-h-96` を超え**ウェッジ系はスクロールが必要**（`record/page.tsx:988`） |
| 4. 強度 | 3択から1つ | 1 | 自動遷移 |
| 5. 風 | 方向8択→中央で強弱2択 | 2（無風は1） | 方向タップ後、強弱タップで遷移（`record/page.tsx:1130-1219`） |
| 6. 結果 | 着弾点を円グリッドにタップ | 1 | ドラッグ微調整可（`record/page.tsx:1281-1334`） |
| 〃 | ターゲット距離をキーパッド入力 | 桁数分（150Yd=3） | 自動デフォルトなし・前打引き継ぎなし（`NumberKeypad.tsx`） |
| 〃 | 保存ボタン | 1 | `record/page.tsx:1490-1498` |

- **合計: 風あり・3桁距離で11タップ／無風・2桁距離で9タップ**。依頼者提示の「約10タップ」はコード上も正確。
- ミスショット打：着弾点の代わりにミスタイプ1タップ（＋距離を入れるなら3タップ）。8〜11タップ。
- 初心者プリセット（傾斜・クラブ・結果のみ）でも、傾斜1＋クラブ1＋着弾1＋距離3＋保存1＝**7タップ**。「最小」でも7。
- 所要秒数: 実測値はなし（**未確認**）。構造からの推定では、6画面の認知判断（8択→6択→17択→3択→16択→連続値＋3桁）を伴うため1打20〜40秒とみられる。タップ数と選択肢数は実測、秒数は推定であることを区別されたい。
- ラウンド換算（90打・全打フル記録）：**約900〜1,000タップ、累積操作30分超**。これは「記録するか、プレーに集中するか」の二者択一を毎打強いる量である。

### 1.2 記録フローの構造的問題

個々の画面は工夫されている（1画面1タップ・自動遷移・大きなタッチターゲット）。問題はパラダイムにある。

1. **全項目×全打×直列強制**。「次へ」ボタンは該当フィールド選択済みのときしか表示されない（`record/page.tsx:1756-1774`）。つまり**有効化された項目はスキップ不能**。風がどうでもいい打でも風を選ばないと先に進めず、ユーザーは「無風」をタップして進む。これは「無風だった」という**虚偽データの混入装置**になっている。惰性で入力された条件データは分析の信頼性をむしろ下げる。
2. **コア入力が最後・最深部にある**。分析の核心（クラブ・着弾点・距離）に到達する前に、サブ条件4画面を通過させる順序設計。疲れて離脱した場合に残るのは価値の低い方のデータである。
3. **賢いデフォルトが皆無**。保存後のリセット（`record/page.tsx:631-649`）でクラブ・距離・条件は毎回 null に戻る。直前のクラブ引き継ぎも、クラブ別平均距離からの距離推定もない。コース・気温だけは引き継ぐが、それは§1.5の通り別のバグを抱えている。
4. **入力途中の状態が揮発する**。currentShot は zustand のメモリのみで persist していない（`lib/store/index.ts` に persist middleware なし）。ラウンド中に PWA がバックグラウンドで kill されると入力途中の打は消える。
5. **記録の編集はサーバーに反映されない**。編集保存はローカル更新のみで、`record/page.tsx:566` に「TODO: Update shot on server as well (requires PUT endpoint)」とあるが、**PUT エンドポイント自体は既に存在する**（`app/api/shots/[id]/route.ts:61-126`）。配線漏れ。「後から直せばいい」という運用が現状では成立していない。

### 1.4 背景説明と実装の食い違い（重要）

依頼の背景に「実装済み」とある2点は、コード上は以下の状態である。

1. **「傾斜はジャイロで自動判別」→ 自動入力はされない**。ジャイロは検出値をハイライト表示するだけで、`useGyro.ts:117-119` に「currentShot.slope is only set when user taps a slope segment / Gyro data is stored in gyro.slope for highlighting only」と明記されている。傾斜タップは省略されない。
2. **「オフライン記録→オンライン復帰で自動同期」→ 実質壊れている**。
   - Service Worker の同期ハンドラは登録されているが、本体 `syncShots()` が**空関数**（`public/sw.js:349-352`、コメント「この部分は後でクライアント側から呼び出される」）。Background Sync API 対応ブラウザ（Android Chrome 等）では、オンライン復帰時に sync イベントが発火しても**何も起きない**。
   - SyncManager 非対応ブラウザ（iOS Safari）はフォールバックで `lib/sync.ts` の `syncShots()` が即時実行される（`lib/sync.ts:16-19`）。しかしこの関数は IndexedDB の**全ショット**（オンライン保存済みで serverId を持つものも含む）を `POST /api/shots` に再送し、成功した分を**ローカルから削除**する（`lib/sync.ts:25-92`）。POST 側に重複排除はない（`app/api/shots/route.ts:49-70` は無条件 create）。
   - 帰結: **iOS ではオンライン復帰のたびにサーバーへ重複レコードが量産され、かつローカルから一時消失**（Layout の `syncShotsFromServer` が再取得するまで分析に出ない。再取得は createdAt 一致で照合するため、サーバー側で増えた重複はそのまま全部ローカルへ戻る）。Android では**オフライン分が同期されない**。
   - これは新UI設計の前提を崩す本番品質問題であり、§5 Phase 0 で最優先修正とする。

### 1.5 その他、記録データの信頼性を毀損している実装

- **距離未入力ショットのサイレント欠損**。保存ボタンは result か missType があれば押せる（`record/page.tsx:1494`、distance 不要）。一方サーバー API は distance 必須で 400 を返す（`app/api/shots/route.ts:41-46`）。クライアントは 400 を catch して「IndexedDB のみ保存」にフォールバックし、ユーザーには**「保存しました」と成功表示**する（`record/page.tsx:607-611, 619`）。この打は再送しても 400 のため**永遠にサーバーへ上がらない**。
- **位置・気温キャッシュに有効期限がない**。`lastLocationData` は timestamp を保存しているのに、読み出し時に一切チェックせず無条件で使う（`record/page.tsx:391-437`）。**1ヶ月前の別ゴルフ場・別季節の気温が今日のショットに自動付与される**。小さな「再取得」ボタンに気づかない限り誤データが蓄積する。コース・気温の「自動取得」は実際には「初回取得値の永久使い回し」である。
- **feeling は幽霊フィールド**。スキーマ（Dexie/Prisma 両方）と入力プリセット定義（advanced: feeling: true）には存在するが、記録UIに入力ステップがなく、設定のカスタム項目一覧（`settings/page.tsx:643-651`）にもなく、分析のフィルタUIにも描画されない。**常に null で保存される死にデータ**。
- 設定の入力レベルは IndexedDB＋サーバー同期だが、Prisma 側の `inputLevel` デフォルトは `"standard"`（`schema.prisma:299`）でクライアントの `'advanced'` と不一致。実害の有無は**未確認**だが、命名のズレは移行時に整理すべき。

### 1.6 「あまり使われていない」原因の仮説

記録負担（上記）に加え、コードから読み取れる複合要因：

1. **デフォルトが最大負荷**。初期値「上級者＝全項目」。負担を減らす設定（初心者プリセット）の存在に、設定ページへ行かない限り気づけない。最も離脱しやすい初回ラウンドを最重負荷で迎える設計になっている。
2. **トライアル制限と価値実感のタイミングが噛み合っていない**。トライアルは「ユニーク日付3日分」で新規日付の記録をブロックし、サブスク誘導 confirm を出す（`app/api/trial/validate/route.ts:63, 89-97`）。月1ゴルファーなら3ヶ月分。クラブ別精度や傾向分析はデータが溜まって初めて面白くなる設計なのに、**溜まる前に課金壁が来る**。「トライアル中心であまり使われていない」のは、記録負担と課金タイミングの掛け算と考えるのが自然。
3. **記録した瞬間のリターンがゼロ**。保存→トースト→ステップ1に戻るだけ。「今の7I、あなたの平均より8Yd右」のような即時フィードバックがなく、記録は将来の分析のための義務でしかない。行動が強化されない。
4. **同期不具合の体感**（iOS でデータが一時消える／重複が見える 等、§1.4）が信頼を削る。
5. オンボーディング導線（`app/learn`、`SettingsGuideModal`、`PwaInstallBanner`）の中身は**未確認**のため、ここの寄与は評価できない。

### 2.2 コア指標（無いと分析が成り立たないもの）

1. **club** — 全集計の主軸
2. **result {x,y}** または **missType**（排他） — 散布図・精度・ミス分析の本体
3. **distance（ターゲット距離）** — 距離帯分析と「平均飛距離」算出に使用。ただし注意：アプリは飛距離を直接記録しておらず、`狙い距離 + 縦ズレy` で推定している。コアなのは「ターゲット距離」であり、これは**クラブ別実績からの自動デフォルトで代替可能**（手入力必須である必要はない）
4. **date / ラウンド紐付け** — 時系列トレンドとフィルタの基盤

### 2.3 サブ軸（あれば嬉しいが、毎打必須にする根拠が現状ない）

- **slope / lie / wind** — 手動フィルタ専用。クロス集計も自動インサイトも未実装。6ステップ中3ステップを費やして集めたデータの出口が「チェックボックスで絞り込めること」だけ、というのが現状の実態。ただしアプリのコンセプト（条件×精度）の根幹でもあるため、**廃止ではなく「例外時のみ記録」へ移行**するのが正しい（§3）。条件は「意図して付けた打」だけに付く方が、フィルタ結果の信頼性はむしろ上がる。
- **strength** — フィルタ専用かつ主観値で再現性が低い。**廃止候補**（オーナー判断事項）。
- **temperature** — 打単位で変える意味がほぼない。実装上も自動取得値の使い回しで実質ラウンド定数。**ラウンド属性へ移動**。
- **golfCourse / 緯度経度 / actualTemperature** — 同上、ラウンド属性へ。

### 2.4 死んでいるデータ

- **feeling** — 入力UIなし・分析UIなし・常に null（§1.5）。正式廃止を提案。
- **manualLocation** — 位置の後追い更新のための運用フラグ。ラウンド単位管理にすれば不要。

### 2.5 依頼者仮説の検証結果

仮説「核心はクラブ×着弾点のズレ×飛距離×ミスの種類。傾斜・ライ・風・気温は絞り込みのサブ軸」は、**分析コードで裏付けられた**。修正点は2つ。

1. 「飛距離」は記録値ではなく `ターゲット距離 + y` の推定値。コア入力は「ターゲット距離」であり、自動デフォルト化できる。
2. 気温はサブ軸ですらなく、実装挙動として「ラウンド定数」。打単位入力から外して問題ない。

## 3. 新記録システムの設計

### 3.1 設計原則

1. **打った直後はコア4（クラブ・着弾点 or ミス・距離・打った事実）だけを最小タップで確定する。** それ以外は記録の前提条件にしない。
2. **デフォルトは「何もしないこと」**。平坦・普通ライ・無風はタップで申告させず、例外時のみ付与する。未操作は「未記録(null)」として保存し、虚偽の「無風」を作らない。
3. **ラウンドという文脈を導入し、打単位から定数（コース・気温・日付）を排除する。**
4. **保存ボタンを廃止し、最後の必須タップ＝保存にする**（Undo で安全性を担保）。
5. **記録直後に1行のリターンを返す**（例:「7I 右8Yd ─ 平均より右」）。記録行動を強化する。
6. **詳細は後から**。当日の補完ビューで条件・メモを追記できる（既存の PUT API を配線）。

### 3.2 推奨案：ワンボード2タップ記録 ＋ あとから補完

ステップ遷移を廃止し、記録を1画面に常駐させる。

```
[ラウンド開始]（1日1回）
  └ GPS→コース名・気温を自動確定（オフライン/失敗時のみ手動1画面）
     ※現行 lastLocationData の使い回しを廃止し、ラウンドに紐付ける

[記録ボード]（1画面・遷移なし）
┌──────────────────────────────┐
│ ⛳ 鳴尾GC ・ 21°C(中間期)      │ ← ラウンドバー（タップで編集）
├──────────────────────────────┤
│ [7I] [8I] [9I] [PW] [52]     │ ← クラブ列：直近使用順に並ぶ
│ [56] [DR] [3W] [5U] [▼全部]  │    前回クラブは選択済み状態で開始
├──────────────────────────────┤
│        着弾点サークル          │
│     （狙い基準 ±70Yd）        │ ← タップ＝即保存（ドラッグ微調整可）
│   ターゲット距離: 150Yd ✎     │ ← クラブ別実績から自動。修正時のみタップ

[本日の補完ビュー]（ラウンド後・任意）

     → サーバーへ PUT 同期（既存APIを配線）
```

操作フロー：
- **通常打: クラブ1タップ → 着弾点1タップ ＝ 2タップで保存完了**。同一クラブ連続なら着弾点のみの**1タップ**。
- **ミス打: クラブ1 → ミスボタン1 ＝ 2タップ**。
- **条件が特殊な打: ＋1〜2タップ**（チップ→8方向ポップオーバー等。ジャイロ検出値があれば傾斜チップに候補表示し、タップ1回で確定）。
- **距離を直したい打: ＋1〜4タップ**（キーパッドはモーダルで existing `NumberKeypad` を流用）。
- ラウンド開始: 1タップ（自動取得成功時）。

得る・失うの明細：

| | 内容 |
|---|---|
| 得る | 記録の網羅性（全打記録が現実的になる）／distance の欠損ゼロ化（自動デフォルト）／ラウンド文脈（コース・気温・日付の正規化、§1.5 のキャッシュ汚染解消）／条件データの信頼性（付いている＝意図的に付けた）／保存即時フィードバックによる継続動機 |
| 失う | 全打悉皆の slope・lie・wind（→例外打のみのスパースデータになる）／strength（廃止する場合）／気温の打単位粒度（→ラウンド単位。実害なし：現行も実質使い回し） |
| 不変 | 散布図・クラブ別精度・平均飛距離・距離帯分析・ミス分析・メモ──**現行分析の全機能はコア4で成立するため無傷**。過去データもそのまま同じ分析に乗る |

「簡単さ→記録数増→分析の質向上」の根拠：現行分析はすべて平均と分布であり、**サンプル数がそのまま精度**になる。1ラウンド10打しか記録されない6次元データより、80打記録される4次元データの方が、クラブ別精度・距離帯・ミス傾向の statistical power は確実に高い。条件分析も「例外時に意図して付けた打」だけが対象になるため、惰性入力の「無風・普通」ノイズが消えてむしろ締まる。

### 3.3 代替案A：ホールまとめ記録（カート内一括）

打った直後は何もせず、ホールアウト後にカートで1ホール分（平均4〜6打）をまとめて入力する。ホールビューに「クラブ→着弾点」を打順で連続入力（1打あたり2〜3タップ、ホールあたり約15秒）。

- 利点: プレー中の操作ゼロ。打数（スコア）記録と自然に統合できる。
- 欠点: 記憶頼みで着弾点精度が低下する（特に1番ホールから数打前の記憶）。ホールアウト直後はグリーン周りで最も忙しく、「カートに戻ってから」の習慣化に賭けることになる。記録忘れ→まとめて再現不能、のリスク。
- 評価: 単独案としては非推奨だが、推奨案の「補完ビュー」をホール単位に拡張する形で**吸収可能**（推奨案でミスだけ打っておき、ホール後に着弾点を埋める運用が自然に成立する）。

### 3.4 代替案B：最小改修（現行ステップUIの調整）

パラダイムを変えず、(1) 初期プリセットを「初心者」に変更、(2) ステップ順を クラブ→結果→（任意で条件） に並べ替え、(3) 距離の自動デフォルトとクラブ引き継ぎ、(4) 保存ボタン廃止（着弾タップ＝保存）。

- 1打: クラブ1＋着弾1＝2〜3タップまで下げられる。実装コスト最小・リスク最小。
- 欠点: 画面遷移パラダイムが残る（着弾→次打のクラブ選択に戻る遷移）。条件の例外付与が「設定で項目ONに戻す」操作のままで、打単位の柔軟性がない。1.2で挙げた構造問題（途中状態の揮発、スキップ不可）が温存される。
- 評価: 推奨案の工数が許容されない場合の次善。ただし「ゴッソリ変えてよい」というビジネス判断が出ている以上、ここで止める積極的理由はない。

|---|---|---|---|---|---|

### 3.6 音声入力を主軸にしない理由

着想例にあった音声入力（「7番、右に10、つま先上がり」）は検討したが、主軸採用は見送りを推奨する。(1) iOS Safari の音声認識はサーバー処理でオフラインファースト制約と衝突、(2) 同伴者の前で発話する心理コスト、(3) 風・屋外騒音下の認識率、(4) 誤認識の訂正UIが結局タップUIと同等に必要。推奨案の構造（コア4の即時記録）の上に、将来の**補助入力**として載せる位置付けが妥当。

## 4. データ構造への影響と移行計画

### 4.1 スキーマ変更（追加のみ・破壊的変更なし）

Prisma（サーバー）:

```prisma
model Round {
  id                String   @id @default(cuid())
  userId            String
  date              DateTime
  golfCourse        String?
  latitude          Float?
  longitude         Float?
  temperature       String?   // summer / mid-season / winter
  actualTemperature Float?
  createdAt         DateTime @default(now())
  updatedAt         DateTime @updatedAt
  user  User  @relation(fields: [userId], references: [id], onDelete: Cascade)
  shots Shot[]
  @@index([userId, date])
}

model Shot {
  // 既存フィールドは全て維持（過去データ互換）
  roundId    String?  // 追加・nullable
  clientId   String?  @unique  // 追加: クライアント生成UUID（同期の冪等キー）
  holeNumber Int?     // 追加・nullable（将来のホール文脈用）
  distance   Int?     // 必須 → nullable へ変更（§1.5 のサイレント欠損の根治）
  ...
}
```

- `distance` の nullable 化は分析コードと互換：analysis / dashboard とも `distance !== null && distance > 0` ガード済み（`analysis/page.tsx:146` ほか）を確認。
- 旧クライアント（必須 distance を送る）はそのまま動く。新クライアントは自動デフォルト値を入れるため、実運用上の null は稀。
- IndexedDB は Dexie `version(6)` で `rounds` テーブルと `shots` への `clientId / roundId / holeNumber` 追加。既存の version(1)〜(5) のアップグレードパス（`lib/db/index.ts:54-93`）と同じ方式で安全に増分できる。

### 4.2 同期の再設計（Phase 0・新UIの前提）

現行の「全件POST→ローカル削除」（`lib/sync.ts`）と空実装のSW同期（`public/sw.js:349-352`）を廃し、以下に置き換える：

1. クライアントは記録時に `clientId`（UUID）を必ず生成。
2. 送信対象は「`serverId` が null のショットのみ」。サーバーは `clientId` unique 制約で upsert（重複時は既存IDを返す）→ **再送しても重複しない（冪等）**。
3. 同期成功後もローカルから削除しない（serverId を書き込むだけ）。分析は引き続き IndexedDB を単一ソースとして読む。
4. 編集 PUT（既存API）を補完ビュー・編集フローに配線。オフライン編集は「dirty フラグ→オンライン時 PUT」。
5. 既存の重複データ対策：`userId × createdAt × club × result` 一致でのサーバー側重複検出スクリプトを用意（実在量は**未確認**のため、まず計測→件数次第で自動/手動マージを判断）。

### 4.3 過去データとの互換

- 既存 Shot は roundId null のまま全分析で従来どおり機能する（分析はフラットな Shot 配列を読むため）。
- 任意のバックフィル：`date × golfCourse` グループから仮想 Round を生成するスクリプト。失敗してもラウンドフィルタが効かないだけで、データは壊れない。
- 条件フィールドの解釈変更（悉皆→例外時のみ）は、**保存値の意味を変えない**（slope='flat' は従来どおり「平坦」、null は「未記録」）。新UIでは未操作打を null で保存する。現行データに含まれる惰性の 'flat'/'none' と新データの null が混在するが、フィルタは値一致なので動作は壊れない。分析画面に「条件記録あり N打」の注記を出すことを推奨。
- CSVエクスポート/インポート（`settings/page.tsx:433-505`）は列構成維持＋ roundId/clientId 列の末尾追加で互換。

## 6. 残るリスク／オーナーに確認すべき判断事項

### リスク

1. **条件データがスパース化する**。「傾斜×クラブ」の悉皆クロス分析を将来の売りにする構想があるなら、例外時記録では母数が細る。緩和策：補完ビューでの後付け／「この打、傾斜ありました？」の軽いプロンプト（ただし通知過多は本末転倒）。
2. **着弾点サークルの精度限界は残る**（±70Yd・タップ精度）。記録数が増えるぶん平均値の信頼性は上がるが、1打単位の精度は上がらない。
3. **新旧データの混在**により、条件フィルタの母数が時期で不均質になる（§4.3 の注記で対処）。
4. **既存の重複・欠損データ**の実在量が未計測。Phase 0 の計測結果次第でクリーニング工数が変動する。
5. ワンボードUIは1画面の情報密度が高い。直射日光・グローブ下での実機検証（Phase 1 のβ目的に含める）が必須。

### オーナー判断事項

1. **トライアル制限の再設計**（最重要）。記録が簡単になると「3日分」は速く消費される。一方で分析の価値はデータ量に比例する。制限軸を「日数」から「分析機能の制限（記録は無制限・高度分析は有料）」へ変える選択肢を検討すべき。記録ハードルを下げて課金壁を記録側に残すのは自己矛盾になる。
2. **strength（強度）を廃止してよいか**。分析実装はフィルタのみ・主観値。残すなら条件チップの1つに格下げ。
3. **feeling の正式廃止**（現状すでに死んでいる：§1.5）。
4. **Round 概念の導入範囲**。ホール番号・打数まで持つと事実上のスコア管理アプリに踏み込む。今回は roundId＋holeNumber(任意) までとし、スコアカード機能は別判断としたい。
5. **保存即時フィードバック（平均との比較1行）の採否**。計算は既存ロジック流用で軽いが、「余計な表示」と感じる層もいる。β でON/OFF検証。
6. **Phase 0（同期修正）を新UIに先行させることの承認**。ユーザーに見える新機能ではないが、これを飛ばして記録数を増やすと、壊れた同期の上で重複・欠損が拡大する。
7. 効果測定の計測基盤（イベントログ）の有無と、なければ最小限の導入可否。
